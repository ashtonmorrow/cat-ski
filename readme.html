A love letter to Windows 3.0 SkiFree, built one weekend.

If you grew up on Microsoft Entertainment Pack 3, you remember it: an endless downhill ski run, dodging trees, until the abominable yeti shows up out of nowhere and eats you no matter how good you've been. The yeti was 1991's most iconic boss fight — undefeatable, sudden, terrifying. I wanted to recreate that feeling but with a cat, ship it as a free thing on the web, and use it to learn how far I could push a vibes-driven build with Claude.

Built by Mike Lee, aka Whisker Leaks. Say hi.

HOW TO PLAY

You're a cat — pick a color — skiing down a mountain — pick a range. Use arrow keys or WASD to steer; down tucks (faster), up carves (slower). Hit jump ramps to launch into the air; press 1, 2, or 3 mid-air to perform tricks (Meow Roll, Whisker Twist, Cat Nap). Chaining tricks across a single jump compounds the bonus. Grab flags as you pass them.

After a fixed time, dogs start chasing you down the slope. They cross your screen-Y at a deterministic moment — you can dodge horizontally, OR launch off a gold escape ramp at the right moment to be airborne when the dog tries to grab. Survive all the dog waves and the snowman appears. The snowman is the SkiFree yeti tribute — instant kill, no escape. Difficulty is fully tunable via two sliders (intensity 0.8×–5×, dog count 1–10) plus four named presets that snap to canonical values.

WHY WE BUILT IT THIS WAY

The Stack

The whole game is a single self-contained HTML file, ~140 KB. No build step, no framework, no dependencies besides two Google Fonts. Every sound is synthesized at runtime via Web Audio's OscillatorNode and filtered noise — not a single audio file. Every sprite is drawn from ctx.fillRect rectangles — no image assets except the social preview and the favicon set. This is partly a tribute to the original SkiFree (60 KB on a 1991 floppy) and partly because the simplest possible stack means iteration is instant: git push, see it live in 30 seconds, iterate.

Hosted on Vercel from a public GitHub repo, custom domain via a single CNAME. PWA-installable with a service worker that caches the app shell for offline play. Total moving parts you have to understand to maintain it: HTML, CSS, vanilla JS, plus 30 lines of service worker. That's it.

Designing each element

Before writing code for a feature, the question was always: what's the criteria? What does this need to do, look like, feel like? A few examples of how that played out:

  • Cat sprite: must read as a cat at 16 px wide, must support four color variants without forking the drawing code. Solved with procedural arcs and ellipses parameterized by a palette object — one drawing function, four palettes, four cats.
  • Dog chase: must be escapable with skill but not by brute-force tucking forever. Solved by giving the dog a constant downward screen velocity, so it always reaches the cat at a calculable moment — the player gets exactly one timing window to dodge or launch. The bonus escape ramps spawn ahead of the player at a distance computed from current player speed and dog descent rate, so the launch window lands perfectly regardless of how fast you happen to be moving.
  • Difficulty: casual players want presets, tinkerers want sliders. Solved with named presets (EASY, MEDIUM, HARD, INSTA DEATH) that snap to canonical slider positions, plus continuous intensity (0.8× to 5×) and dog count (1–10) sliders. Pick a preset, fine-tune from there.

Color, derived from real photographs

The first pass at mountain themes used default-AI palettes — a generic pinky-cream "Rockies", a default cool-blue "Alps". They felt synthetic. To make each region actually read as itself, I described the visual character of each range from photography rather than asking for "warm" or "cool":

  • Alps: cool crisp blue snow, deep emerald Norway spruce, granite-gray peaks fading blue with atmospheric haze.
  • Rockies: warm afternoon light on sandstone and exposed granite, slightly bluer-green lodgepole pine, rust-toned distant peaks — think golden hour in Estes Park.
  • Himalayas: deep saturated alpine blue (less atmosphere = more saturation), brilliant white snow, near-black slate rock, sparse dusty juniper at low elevations only.

That description-first approach extended to the obstacles. Norway spruce silhouettes for the Alps, lodgepole pine (taller and narrower with a visible trunk) for the Rockies, krummholz juniper (stunted, wind-shaped) for the Himalayas. Aspen logs — white bark with iconic black eye scars — in the Rockies, because that's the most recognizable Western tree feature. Bare granite boulders in the Himalayas, because at altitude there are no logs. Each species got researched, pitched as a silhouette description, and drawn as ~30 lines of fillRect calls.

A DIY tip if you're trying this with Claude

The single biggest leverage point was treating Claude as a design collaborator, not just a code-writing tool. Three questions before any feature:

  • What's the criteria? What does this need to do, look like, feel like — before any code.
  • What's the simplest implementation? Often this surfaced "you don't actually need that" answers and saved a lot of code.
  • What are the trade-offs? Procedural sprites mean no asset pipeline but harder to iterate visual detail. Web Audio means no audio files but the sounds are synthetic. Knowing the trade-off lets you pick consciously.

For tasks like color palettes — instead of asking Claude to "make a Rockies theme," I described light, geology, and vegetation in plain words ("warm afternoon light on sandstone and exposed granite") and let Claude derive the hex codes that capture that feel. For the icons, I dropped a square reference PNG into the repo and Claude produced the multi-resolution favicon set in one prompt. For sounds, I described what each event should evoke ("dog double-bark for a dramatic entrance"; "trick chain that ascends in pitch as the chain grows") and Claude built the OscillatorNode envelopes.

What made the loop work: every iteration was deployable in seconds (no build step), so the cycle was push → look at the live site → iterate. Across 50+ commits in a few days, the "ship and verify" cadence mattered more than any individual code decision.
